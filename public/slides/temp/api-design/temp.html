Sourcing design artifacts > Database models

Database models are structured representations of the data architecture for a database.

They define how data is stored, organized, and manipulated within a database system.

There are several types of database models. For our purposes, we will focus on the Entity-Relationship (ER) Model.

Structure

Data is represented using entities (objects) and relationships between those entities.

Example

A school database where students, teachers, and classes are entities, and the relationships define which students are in which classes and which teachers teach which classes.

Distinct existence

Entities have their own identity and can be uniquely distinguished from other entities.

Example: Each student in a school database is a unique entity.

Entity types

Entities are grouped into entity types, which define a set of entities with the same attributes.

Example: All students in a school form the "Student" entity type.

Primary Key

Each entity type has a primary key, an attribute (or a combination of attributes) that uniquely identifies each entity instance. Example: The student_id attribute in the "Student" entity type serves as the primary key.

Relationships

Entities can be related to other entities through relationships.

Example: A "Student" entity might be related to a "Course" entity through an "Enrollment" relationship, indicating that a student is enrolled in a course.

Entity-Relationship (ER) diagrams

They provide a visual representation of the entities (or tables) in the database and the relationships between them.

Cardinality specifies the number of instances of one entity that can be associated with the number of instances of another entity. It defines the minimum and maximum number of occurrences of one entity that can be linked to the occurrences of another entity in a relationship. Cardinality is often indicated using symbols such as "1" (one), "M" (many), or "*" (zero or more).
Relationship types

Understanding the types of relationships and how they are depicted in an ER diagram is essential for designing a robust and normalized database schema.

  • One-to-One (1:1)

    Definition: Each entity associates with only one entity in another set.

    Example: Each patron has one unique loyalty card; each loyalty card is tied to only one patron.

    Notation: PATRON ||--|{ LOYALTY_CARD

  • One-to-Many (1:N) or Many-to-One (N:1)

    Definition: One entity in a set links to multiple entities in another set.

    Example: A patron places multiple orders; each order belongs to one patron.

    Notation: PATRON ||--o{ ORDER

  • Many-to-Many (M:N)

    Definition: Multiple entities in each set relate to multiple entities in the other set.

    Example: Dishes appear in multiple menus; each menu includes multiple dishes. Usually resolved with a junction table like MENU_DISH.

    Notation: DISH ||--|{ MENU
