Web API basics

What are APIs?

APIs enable software applications to communicate and exchange data by defining interaction rules and protocols.

What is a web API?

Web APIs allow web apps to get data from external sources over the internet, like a travel site fetching flight deals from airlines.

What are REST APIs?

REST APIs use standard HTTP methods (GET, POST, PUT, DELETE) to interact with resources via URLs, ensuring scalability and simplicity.

REST API client-server sequence
Other API types

Course Focus

We'll focus on REST APIs, but the documentation techniques apply to other API types too.

API lifecycle roles

There are three primary API lifecycle roles:

API Consumers

Use APIs to access data or services from other systems, like developers integrating third-party services or businesses for data analysis.

API Producers

Create and maintain APIs, ensuring they meet consumer needs and industry standards.

API Designers

Specialize in designing APIs, focusing on structure, endpoints, and usability.

Internal vs. external APIs

External APIs
  • External APIs are public, used by developers, vendors, and customers to build applications and interfaces.
  • They generate revenue, enhance brand visibility, manage backend interactions, and integrate with other systems, with usage metrics collected for improvements.
Internal APIs
  • Internal APIs are private, used within a company to access sensitive data and boost operational efficiency.
  • They focus on specific tasks, connect backend services, and support microservices architecture, facilitating internal communication and development.

API business sectors

APIs are used across a variety of business sectors, for example:

  • E-commerce: Integrate with payment gateways, manage inventory, process orders, retrieve product info.
  • Finance and Banking: Access account info, process transactions, integrate payment systems, provide financial data.
  • Healthcare: Facilitate data exchange between healthcare systems, manage patient data, EHRs, telemedicine.
  • Government and Public Services: Access public data, government services, civic information.
  • Education: Manage student information, learning platforms, online content, academic analytics.

APIs vs. SDKs

Quick recap: What are APIs?

Define how software systems interact, establishing rules and protocols.

What is an SDK?

Comprehensive toolsets including APIs, tools, libraries, docs, and sample code, aiding app development for specific platforms or frameworks.

How are they different?

APIs specify interaction methods, while SDKs provide complete development toolkits.

API documentation
common issues

Observational Study
  • An observational study, "How developers use API documentation" (Meng et al., August 2019), explored if API documentation issues stem from a mismatch with developer habits.
  • The study involved an eCommerce REST API and participants with varying experience levels from diverse organizations, who had no prior exposure to the API.
  • Developers completed five tasks in 40 to 70-minute sessions using the provided documentation.

Research Findings

The study found that task-based documentation is more effective than arbitrary content organization, aligning better with developers' problem-solving approaches. Key observations included:

  • Incomplete OpenAPI descriptions hinder understanding and usage.
  • Documentation often relies on arbitrary content signals rather than logical, task-based organization, mismatching developer habits.
  • Poor navigation elements make finding information difficult.
  • Incomplete code samples hinder usability; redundancy in samples and content is crucial.
  • Lack of a clear connection between code and content, with code comments preferred by opportunistic developers.
  • Conceptual information should be quick and practical, as detailed theory is less favored.

Introduction to OpenAPI

What is OpenAPI?

According to SmartBear, the parent company of Swagger:

“The OpenAPI Specification (OAS) defines a standard, language-agnostic interface to RESTful APIs which allows both humans and computers to discover and understand the capabilities of the service without access to source code, documentation, or through network traffic inspection.”

Let's break it down.

Defines a standard

OpenAPI specifies the structure and description of an API.

Language-agnostic interface to RESTful APIs

OpenAPI works only with REST APIs using HTTP, enabling cross-platform interaction.

Allows both humans and computers to discover and understand service capabilities

OpenAPI generates readable documentation and defines request/response structures for client-server communication.

Without access to source code, documentation, or network traffic inspection

OpenAPI separates client applications from the API server, allowing interaction based on the API definition alone.

In summary

OpenAPI is a RESTful API specification that describes APIs, enabling both humans and computers to understand and interact with them.

What is OpenAPI written in?

OpenAPI specifications can be written in JSON or YAML, both popular for their readability and ease of use in data representation and configuration.

Writing JSON is crucial since request bodies need to be composed in JSON format.

JSON tutorial

What is JSON?

JSON (JavaScript Object Notation) is a lightweight, easy-to-read format for data exchange, widely used for transmitting data in web applications.

Key Features of JSON
  • Human Readable: JSON is designed to be easily readable by humans.
  • Lightweight: JSON has a minimal syntax, making it an efficient format for data exchange.
  • Language Independent: JSON is supported by many programming languages, making it a universal format for data exchange.
Common Uses of JSON
  • Configuration Files: JSON is often used to store configuration settings.
  • Data Exchange: JSON is used to transmit data between a server and web application.
  • APIs: Web APIs use JSON to communicate with clients.
JSON Structure

JSON is built on two structures:

  • Objects: A collection of key/value pairs.
  • Arrays: An ordered list of values.
JSON Syntax Rules
  • Data is in name/value pairs: A name (or key) is followed by a colon and then the value.
  • Data is separated by commas: Name/value pairs are separated by commas.
  • Curly braces hold objects: Objects are enclosed in curly braces {}.
  • Square brackets hold arrays: Arrays are enclosed in square brackets [].
Validating JSON

It's essential to validate JSON data to ensure it is well-formed. Tools like JSONLint can help you validate and format JSON data.

Servers and paths

Full resource URLs

A full resource URL pinpoints the exact location of a resource or endpoint on a web server.

It's formed by combining the server URL with the endpoint path specified in the OpenAPI spec.

servers object

The servers object lists one or more base URLs for the API.

These URLs are the foundation for all endpoints defined in the paths object.

Each URL can have a description to clarify its purpose, such as production, development, or staging.

Example of OpenAPI servers Object

In an OpenAPI specification, you might define these different servers as follows: 

openapi: 3.0.0
info:
  title: Bookstore API
  version: 1.0.0
servers:
  - url: "http://localhost:8080"
    description: Local development server
  - url: "https://staging.bookstore.com/api/v2"
    description: Staging server
  - url: "https://bookstore.com/api/v2"
    description: Production server
paths: {}
Types of Servers

Let's take a closer look at the different server types:

Development
  • URL Example: http://localhost
  • Purpose: For developers to build and test code on their local machines.
Staging
  • URL Example: https://staging.bookstore.com/api/v2
  • Purpose: For testing and quality assurance, mimicking the production environment.
Production
  • URL Example: https://bookstore.com/api/v2
  • Purpose: The live server for end-users, optimized for reliability, performance, and security.

Only reference the production server URL in documentation!

Analyzing a server URL

Let's break down an example server URL to understand how it's constructed:

https://bookstore.com/api/v2

Protocol
  • Example: https://
  • Purpose: The protocol (HTTP and HTTPS) for accessing the API.
Host
  • Example: bookstore.com
  • Purpose: Specifies the domain name of the API server. It indicates where the API is located.
Base Path
  • Example: /api/v2
  • Purpose: Defines the root path for all API endpoints. This is appended to the host to form the base URL. Typically includes the API version.
Paths

The paths object specifies the endpoints (or paths).

These paths are appended to the base URL from the servers object to form complete API request URLs.

An endpoint is a specific path added to the base URL to access a particular resource within the API.

Example path (i.e., endpoint)
openapi: 3.0.0
info:
  title: Bookstore API
  version: 1.0.0
servers:

Collection endpoints

Definition

Collection endpoints are used to retrieve or manipulate lists of resources.

Resource vs. resource instance

A resource represents a concept or an entity that can be accessed or manipulated, such as users, orders, or products.

A resource instance is a specific example of that resource.

For example, in a library system:

  • Resource: Books
  • Resource Instance: A specific book, like "1984" by George Orwell

Resources follow a schema, which defines the set of properties that each resource instance can have.

Filtering

Query parameters, known as "filters" allow clients to customize response data.

Two common filter types are:

  • projection filters: specify which fields to include.
  • selection filters: define criteria to filter results.

Clients can use projection filters to specify which resource properties to include in the response using a comma-separated list.

A common name for a projection is fields, since you are specifiying which fields to include in the response.

Projection filters sometimes support bracket notation [] to access nested resource properties.

Projection filters, like all query parameters, are defined in the parameters section of an operation in an OpenAPI spec: 

parameters:
  - name: fields
    in: query
    description: |
      Projection filters for including only certain resource
      properties in the response. The value for the parameter
      should be a comma-separated list of properties from the
      schema. Supports bracket notation to access nested
      properties.

Let's imagine a scenario where there is a user resource.

The user resource may contain a number of fields, but you may only need a subset: id, name, and the city field from address.

To only include the desired fields in the response, you would construct following projection filter: 

fields=name,address[city]

Then, you can compose and execute a cURL request using the projection filter: 

curl -X GET "https://api.example.com/users?fields=name,address[city]"

Just for fun, let's add a sorting parameter to also sort the results by the name field: 

curl -X GET "https://api.example.com/users?fields=name,address[city]&sort=name:asc"

The resource instances returned in the response only include the specified fields and are sorted by name! 

{
  "results": [
    {
      "id": 2,
      "name": "Jane Smith",
      "address": {
        "city": "Los Angeles"
      }
    }
    {
      "id": 1,
      "name": "John Doe",
      "address": {
        "city": "New York"
      }
    },
  ]
}

The other type of filter is the selection filter.

API consumers can use selection filters to filter resources based on provided criteria.

Selection filter design varies by API.

Let's imagine there's an endpoint that retrieves a list of products: GET /products.

We want to filter the list of products returned by the endpoint.

To do this, we can specify filters using the field name, followed by a dot (.), a comparison operator, and the filter value.

In this hypothetical example,
the supported comparison operators include:

  • lt: (less than)
  • lte: (less than or equal to)
  • eq: (equal to)
  • gt: (greater than)
  • gte: (greater than or equal to)

Let's construct a selection filter to return products created on or after April 23, 2024, and priced less than 8 currency units. 

select=created_at.gte:2024-04-23,price.lt:8

Then, you can compose and execute a cURL request using the selection filter: 

curl -X GET "https://api.example.com/products" \
  -G \
  --data-urlencode "select=created_at.gte:2024-04-23,price.eq:8"

When you use the --data-urlencode options with the -G flag in a multi-line cURL request, cURL appends the encoded parameters to the URL.

The resource instances returned in the response only include the fields that match the specified criteria.

Like projection filters, some APIs have selection filters that support bracket notation to access nested resource properties. For example: 

curl -X GET "https://api.example.com/products" \
  -G \
  --data-urlencode "select=created_at.gte:2024-04-23,price.eq:8,manufacturer[location].eq:USA"
Pagination

Pagination is the process of dividing large datasets into smaller, manageable parts called pages.

It's commonly used in APIs to retrieve and display data in a structured and organized manner.

There are two types of pagination:

  • Page-based Pagination: Uses limit and offset parameters.
  • Cursor-based Pagination: Utilizes limit parameter alone.

In both pagination types, the limit parameter specifies the maximum number of results to include in each page of the response.

For page-based pagination, the limit parameter is used in conjunction with the offset parameter.

The offset parameter determines the starting point from which to retrieve results before displaying results.

This allows API consumers to skip a certain number of records before fetching the subsequent page.

Once the offset is applied, the API returns results, restricting the count to the value specified in the limit parameter.

When offset=0, the API starts returning resources from the very beginning.

For example, executing the following cURL request will limit the number of results per page in the response to 10 (limit parameter), starting from the 21st resource onwards (offset parameter). 

curl -X GET "https://api.example.com/orders?limit=10&offset=20"

The other type of pagination is cursor-based.

In cursor-based pagination only the limit parameter is used.

Like with page-based pagination, the limit parameter limits the amount of results per page, as demonstrated by the following cURL request. 

curl -X GET "https://api.example.com/articles?limit=2"

However, something interesting happens in the response. A cursor value is returned. 

{
  "results": [
    {
      "id": 1,
      "title": "The Importance of Healthy Eating",
      "content": "Lorem ipsum dolor sit amet."
    },
    {
      "id": 2,
      "title": "Exploring Meditation Techniques",
      "content": "Sed do eiusmod tempor incididunt."
    },
  ],
  "cursor": "eyJjb25kaXRpb25UeXBlIjoibmV"
  "total_results": 20,
}

The cursor parameter indicates the position of the next set of records to retrieve.

The API consumer can pass the cursor value as a query parameter in an API request: 

curl -X GET "https://api.example.com/articles" \
              -G \
              --data-urlencode "limit=2" \
              --data-urlencode "cursor=eyJjb25kaXRpb25UeXBlIjoibmV"

The API server will determine the starting point for displaying results based on the cursor value, then limit the amount of results per page based on the limit parameter (as normal).

Note that the total_results field returned in the response typically represents the total number of results in the entire dataset, not just those currently displayed on the page.

Pagination types use cases

Page-based pagination is suitable for the following scenarios:

  • Efficient navigation: Suitable when clients want to navigate through specific pages of results.
  • Static Data: Useful for datasets with stable data where new entries are not frequently added or removed.

Cursor-based pagination is suitable for the following scenarios:

  • Dynamic Data: Appropriate for datasets with frequently changing or dynamic data where the total number of records may vary between requests.
  • Real-Time Updates: Ideal for scenarios where clients need to retrieve the most up-to-date data without missing or duplicating entries.

Creating resources

Definition

The POST method is an HTTP request method used to submit data to be processed by a specified resource.

It is commonly used to create new resources on the server or to submit data for processing, such as submitting a form or uploading a file.

Example use cases
  • Creating a user account
  • Submitting a form
  • Uploading a file
  • Adding a database entry
Security

POST requests are sometimes considered sensitive because they can initiate actions that modify or create data on the server.

To prevent unauthorized or malicious actions, authentication is often required to verify the identity of the user making the request (API keys, tokens, OAuth).

Request bodies

POST requests typically include a request body that contains the data needed to create the resource.

The request body must specify the media type, most commonly application/json, indicating the format of the data being sent.

API security overview

API security encompasses authentication and authorization mechanisms designed to safeguard API resources against unauthorized access and misuse.

API security overview - cont'd

These measures ensure that only authenticated and authorized users or systems can interact with API endpoints, maintaining the integrity, confidentiality, and availability of data.

Authentication vs. Authorization

Authentication verifies the identity of users or clients accessing an API.

Authentication vs. authorization - cont'd

Authorization determines the actions or resources they are allowed to access post-authentication.

Authentication vs. Authorization - cont'd

In short:

  • Authentication: Proves the identity of users or clients.
  • Authorization: Specifies the permissions or access rights of authenticated users or clients.
Authentication methods

APIs employ various methods to ensure the security and integrity of data exchange between clients and servers.

Authentication methods - cont'd

Some methods are primarily used for authentication, while others are commonly used for authentication and authorization.

Authentication methods - cont'd

Authentication Only:

  • API Keys: Unique identifiers for client authentication, lacking fine-grained access control.
  • Basic Authentication: Sends credentials (username/password) for client authentication, potentially insecure due to plain-text transmission.
Authentication methods - cont'd

Both authentication and authorization:

  • Bearer Tokens: Tokens issued post-authentication for subsequent requests, often containing authorization data.
  • JWT (JSON Web Tokens): Compact tokens with encoded claims for both authentication and authorization.
  • OpenID Connect: Authentication layer leveraging OAuth 2.0 for user authentication and single sign-on capabilities.
Authorization for API keys and basic authentication
  • For API Keys and Basic Authentication, the primary function is authentication.
  • Authorization occurs separately based on user roles or permissions assigned on the backend.
Authorization Headers
  • Most authentication methods utilize the Authorization header for API requests.
  • Despite being named "Authorization," this header primarily carries authentication credentials.
  • It may also serve an authorization function, determining permitted actions post-authentication.
Including the Authorization header in requests

The following is an example cURL Request passing a placeholder API key in the Authorization header: 

curl -X GET "http://example.com/api/resource" \
  -H "Authorization: Bearer YOUR_API_KEY"
Error status codes

It's important to differentiate between the HTTP status codes 401 and 403.

  • 401 Unauthorized: Deals with authentication; credentials are required or invalid.
  • 403 Forbidden: Deals with authorization; the server understands but refuses the request.

While HTTP status code 401 is labeled as "Unauthorized," it actually indicates a lack of authentication for the user. 

HTTP/1.1 401 Unauthorized
                Content-Type: application/json
                
                { "error": "Unauthorized. Authentication is required, or the provided credentials are invalid." }

Error response when the API consumer is not authenticated

On the other hand, code 403 is returned when the API consumer is unauthorized to access or perform specific actions on a resource. 

HTTP/1.1 403 Forbidden
                Content-Type: application/json
                
                { "error": "Forbidden. The server understood the request, but it refuses to authorize it. Authentication will not help."}

Error response when the API consumer does not have access to the resource

Summary
  • Some methods serve authentication only, while others provide both authentication and authorization.
  • The Authorization header pertains more to authentication than authorization.
  • A 401 status code signifies authentication failure, while 403 indicates authorization refusal.

Security schemes

Definition

Security schemes in OpenAPI define how security is applied to the API.

They specify the authentication methods and protocols used to secure API endpoints.

Security scheme definitions

A security scheme is defined in the components section of the OpenAPI specification. 

components:
  securitySchemes:
    apiKeyAuth:
      type: apiKey
      in: header
      name: Authorization

Let's break down the security scheme:

  • Type: Specifies the type of security scheme, such as apiKey, http, oauth2, etc.
  • In: Specifies the location of the security token, such as header, query, cookie, etc.
  • Name: Specifies the name of the header, query parameter, or cookie containing the security token.
Global security

When a security is applied globally, it means the security scheme is added to all API endpoints.

Use cases:

  • When all API endpoints require the same authentication method.
  • To enforce consistent security across the API.

A security is applied globally when it is added to the security property at the root level of the OpenAPI document. 

security:
  - apiKeyAuth: []
Operation-level security

Security at the operation level means a security scheme is applied to specific API endpoints.

Use cases:

  • When different endpoints require different authentication methods.
  • To provide flexibility in authentication based on the sensitivity of endpoints.

Security at the operation level can be used in conjunction with global security settings to provide granular control over authentication.

Overview

Cardinality in an ER diagram specifies the number of instances of one entity that can be associated with the number of instances of another entity in a relationship.

It defines the minimum and maximum number of occurrences of one entity that can be linked to the occurrences of another entity.

In ER diagrams, cardinality is usually represented using symbols and annotations on the lines connecting entities.

Cardinality is important for understanding the nature and constraints of relationships in the database schema.

Cardinality notation commonly includes symbols to represent the minimum and maximum occurrences.

One-to-One (1:1)
  • Definition: Exists when each record in one entity is associated with exactly one record in another entity, and vice versa.
  • Notation: ||--||

Example: Each Parrot belongs to only one Pirate, and each Pirate owns only one Parrot.

One-to-Many (1 )
  • Definition: Occurs when each record in one entity is associated with one or more records in another entity, but each record in the second entity is associated with exactly one record in the first entity.
  • Notation: ||--o{

Example: Each Monkey eats multiple Bananas, but each Banana is eaten by only one Monkey.

Many-to-One (N:1)
  • Definition: Occurs when multiple records in one entity are associated with exactly one record in another entity.
  • Notation: ||--o{

Example: Each Astronaut lives on only one Spaceship, but each Spaceship can house multiple Astronauts.

Many-to-Many (M)
  • Definition: Exists when multiple records in one entity are associated with multiple records in another entity.
  • Notation: }o--o{

Example: Each Alien can fly in multiple UFOs, and each UFO can be flown in by multiple Aliens.

Unprotected endpoints

These endpoints do not require authorization, such as an access token or basic authentication using a username and password.

They are primarily used by end-user applications, like food ordering apps, for non-sensitive operations such as retrieving a list of dishes.

An exception exists for placing orders (POST operation), which allows end users to submit orders without authentication.

Protected endpoints

These endpoints are dedicated to administrative functions and necessitate admin-level access to perform CRUD operations (Create, Read, Update, Delete).

Examples include creating ingredients, dishes, and menus.

Admins authenticate via a separate admin application using username and password credentials to access these secure endpoints.

The OAuth ROPC flow

For secure access to protected resources, the course API employs the OAuth Resource Owner Password Credentials (ROPC) Flow.

This flow enables an application to exchange a user's username and password directly for an access token

This ensures robust security measures are in place for administrative functions.

Let's breakdown of the ROPC flow steps.

Client Authentication

The client application authenticates itself to the authorization server using its client credentials (client ID and client secret).

User authentication

The end-user provides their username and password directly to the client application.

Token request

The client application sends a token request to the authorization server's token endpoint, including the user's credentials, client credentials, and relevant parameters (grant type).

Token issuance

The authorization server validates the provided credentials. Upon successful validation, it issues an access token directly to the client application.

Access token usage

The client application utilizes the access token to access protected resources on behalf of the user, ensuring secure and authorized interactions.

Information types

API documentation is generally divided into two main categories: API reference and conceptual documentation.

On an API documentation site, this distinction is often reflected in the information hierarchy.v

Oftentimes, you'll see conceptual documentation labeled "Docs," "Documentation," or "Developer Guides."

These sections typically include the API introduction, key concepts, getting started guides, and more.

However, the term "conceptual documentation" can be overly broad.
This category often includes a variety of content, not just conceptual but also task-oriented (i.e., procedural) information.

For instance, "Getting Started" guides are one of the most important types of information that is unbiquous across any API.

Despite being task-based and focused on helping users perform specific actions, they are often miscategorized as conceptual documentation.

Identifying information types

Effective API documentation involves identifying the different types of information in our API documentation.

By understanding these information types, we can accurately ensure content we create and ensure it meets the specific goals associated with each type.

Instead of creating our own categories, we can borrow concepts from DITA (Darwin Information Typing Architecture).

DITA defines three high-level information types: concepts, tasks, and references.

Let's look at some common examples of these in API docs.

Concepts

Concepts provide background information necessary to understand tasks and reference material. They explain the "what" and "why" aspects of a subject.

Typical conceptual topics include:

  • API introduction: provides an overview of the API, its purpose, and key functionalities.

Typical conceptual topics include:

  • Key concepts and terminology: explains essential concepts and terms used in the API, including data models.

Typical conceptual topics include:

  • Use cases: Describes scenarios in which the API can be effectively used.

Typical conceptual topics include:

  • Resource descriptions: Describes the API's primary entities and their attributes.

Typical conceptual topics include:

  • High-level software architecture: describes the overall architecture and how different components interact.

Typical conceptual topics include:

  • Authentication and authorization methods: details the methods used for securing API access.
Task-based topics

Task-based topics describe how to accomplish specific procedures or actions. They provide step-by-step instructions to help users achieve particular goals.

Typical task-based topics include:

  • Getting started tutorials: Step-by-step guides to help users quickly begin using the API.

Typical task-based topics include:

  • Functionalities & Recipes: Code samples demonstrating syntax and usage for individual or related calls (functionalities), or multiple features for specific use cases (recipes).

Typical task-based topics include:

  • Troubleshooting/Error handling guides: In-depth troubleshooting for specific API flow issues, offering step-by-step solutions for encountered errors.
References

Reference topics provide structured, detailed information about commands, fields, elements, and parameters. They serve as look-up information and are highly navigable.

Typical reference topics include:

  • API reference: A detailed guide to the available endpoints, methods, request parameters, and responses for an API.

Typical reference topics include:

  • Release notes: A record of changes, updates, and fixes made to the API in each version.

Typical reference topics include:

  • Glossary: A list of specialized terms and their definitions used within the API documentation.

Typical reference topics include:

  • Frequently asked questions (FAQs): A section addressing common queries and concerns users may have about the API.

To summarize, thinking about information types in this way helps us to write more effective and purposeful documentation.

Ask, is the purpose of this information to…

  • To explain what something is (Concepts)?

Ask, is the purpose of this information to…

  • To show how to do something (Tasks)?

Ask, is the purpose of this information to…

  • To provide quick references (References)?

Moving forward, the primary content categories in our API documentation will be conceptual documentation, procedural guides, and reference material.

How developers use API docs

With our understanding of the different information types, it’s time to start outlining our documentation.

We are now entering the realm of information architecture.

This discipline involves organizing, structuring, and labeling content.

The goal is to help users find information and complete tasks.

To organize our documentation, we should base our structure on research specific to API documentation.

This involves examining studies on how developers interact with API docs.

This way, we can avoid common pitfalls caused by poor information architecture.

About the study

The research paper "How Developers Use API Documentation - An Observational Study (Meng et al., August 2019)"
will guide our information architecture strategy.

This study suggests that documentation focusing on API functionality is more effective than arbitrary organization.

In essence, API documentation should be structured around the API's functionality to aid developers in completing specific tasks.

Study methodology

The study was observational in nature.

It focused on the eCommerce API, which is based on the REST (Representational State Transfer) paradigm.

The participants were developers from various organizations, including software companies and universities.

They had no prior experience with the API.

Each test session lasted between 40 to 70 minutes.

The participants were asked to solve five predefined tasks using the provider’s documentation.

Research goals

The research goals were to:

  • Characterize strategies used when starting work with a new API.

The research goals were to:

  • Analyze how developers utilize API documentation resources to complete the assigned tasks.

The research goals were to:

  • Identify content-related factors that hinder efficient task completion.

The research goals were to:

  • Assess developers' understanding of API documentation and the time spent on different sections.
Hypothesis

Issues with API documentation may relate to usability and mismatches with developer habits.

Effective API learning requires understanding developers' task-solving strategies and preferred information sources.

Neglecting task-oriented approaches can lead to reliance on subjective content.

The content categories/labels used were:

  • Concepts
  • Integrations
  • Samples

Formal vs. informal OpenAPI elements

The following is an excerpt from Designing APIs with Swagger and OpenAPI by Joshua S. Ponelat and Lukas L. Rosenstock:

The difference between formal and informal descriptions is whether they follow strict rules (a specification). A formal description can be more readily consumed by software, whereas informal descriptions cannot.
Formal elements

Formal elements adhere strictly to predefined formats, schemas, or rules. They are machine-readable and enforce consistency and interoperability.

In the OpenAPI specification, formal elements include paths, operations (HTTP methods), parameters (with defined types and constraints), response structures (status codes, headers, bodies defined using JSON Schema), and security definitions (authentication mechanisms).

The formal elements provide a machine-readable framework that follows strict syntax and semantics. 

paths:
  /pet/{petId}:
    get:
      tags:
        - pet
      summary: Find pet by ID
      description: Returns a single pet
      operationId: getPetById
      produces:
        - application/json
        - application/xml
      parameters:
        - name: petId
          in: path
          description: ID of pet to return
          required: true
          type: integer
          format: int64 
          example: 23
      responses:
        '200':
          description: successful operation
          schema:
            $ref: '#/definitions/Pet'
        '400':
          description: Invalid ID supplied
        '404':
          description: Pet not found
      security:
        - api_key: []

Formal OpenAPI structures and elements are used by machines to validate data exchanged between clients and servers.

Informal elements

Informal elements provide more context, explanation, or interpretation. They may include human-readable text, comments, or annotations that supplement the formal structure.

Within informal elements, humans can use natural, descriptive language without being constrained by rigid formats or schemas to document various aspects of the API.

In the OpenAPI specification, informal elements include descriptions (textual explanations of endpoints, parameters, or responses) and summaries (brief overviews of resources), and field examples.

Informal elements boost readability for humans while adhering to the formal OpenAPI structure.

Unlike formal elements, informal elements are not used to validate data. Their primary function is to provide context to human readers and serve as markup for rendering documentation.

It is crucial to maintain a clear differentiation between formal and informal language in technical documentation.

This practice ensures that information interpretable by machines remains structured and concise, while human-readable content maintains its readability and accessibility.

What is API reference?

API reference documentation is generated from the OpenAPI spec. It describes both formal elements (such as endpoints and data models) and informal elements (such as descriptions, summaries, and examples) of the API.

Unlike conceptual or procedural documentation, API reference ideally covers all OpenAPI elements comprehensively.

If the API reference is generated correctly, it will accurately reflect the formal structure of the API.

Additionally, humans will have documented the individual components of the API in natural language for other humans to read (i.e., informal descriptions were added).

If OpenAPI spec lacks informal descriptions, the resulting API documentation will be incomplete.

For instance, endpoints, parameters, and responses may lack descriptions, which are crucial for providing readers with insight into their meaning.

This deficiency diminishes the clarity and usefulness of the documentation for all readers.

Therefore, formal descriptions alone are not sufficient for complete documentation.

What is Markdown?

Markdown is a lightweight markup language used extensively for formatting plain text documents.

It provides a simple yet effective way to structure content using easy-to-read syntax, making it popular for various types of documentation and online content creation.

Markdown offers several advantages over traditional markup formats commonly used for writing documentation.

Its simplicity and readability make it accessible to users without extensive technical knowledge.

Compared to complex markup languages, Markdown allows authors to focus more on content creation rather than formatting intricacies.

Moreover, its plain-text nature ensures compatibility across different platforms and tools.

Why is learning Markdown important?

Learning Markdown is important due to its widespread adoption across authoring tools and platforms.

It has become the de facto standard for writing developer documentation, including README files and content generated by static site generators.

Many help authoring and knowledge base tools support Markdown, enabling efficient content creation and management.

This versatility makes Markdown an essential skill for anyone involved in software development or technical writing.

It allows documentation to be treated as code, stored alongside project repositories, and managed using version control systems like Git.

Markdown’s relevance to OpenAPI

Markdown's relevance extends to OpenAPI specifications, where it supports the CommonMark standard for fields containing informal descriptions.

This capability enhances the clarity and readability of API documentation by allowing formatted text, including headings, lists, and emphasis.

However, CommonMark is typically not used in fields requiring strict adherence to structured data, such as path definitions or schema properties in OpenAPI.

What is CommonMark?

CommonMark itself is a standardized version of Markdown designed to ensure consistent parsing and rendering across different implementations.

It provides a baseline set of rules that maintain compatibility and predictability, reducing ambiguity in Markdown interpretation across various platforms and tools.

Documenting endpoints

API endpoints are specific URLs (Uniform Resource Locators) that an API exposes to allow clients (such as applications or other services) to interact with it.

For example, you could refer to the endpoint below as the
POST /users endpoint, comprised of a method and a path:

POST

Create a user account

Each endpoint represents a unique functionality or resource that the API provides, typically through HTTP methods like GET, POST, PUT, DELETE, etc.

These endpoints define the structure and behavior of the API, detailing how clients can send requests and receive responses.

Documenting API endpoints is crucial for ensuring clarity, consistency, and effective integration of API functionalities.

Let's now explore some best practices for describing endpoints.

Defining "users" of an API

A common issue is determining how to refer to "users" of an API.

When we think of "users," we often envision a human interacting with a software application, clicking buttons, and submitting form data.

However, the "users" of APIs are entities or systems that interact with the API, such as:

  • web applications
  • mobile apps
  • server-side apps
  • automated scripts
  • IoT devices

Referring to all these possibilities isn't practical, so it's important to choose a consistent term.

There are no universal standards for this, so the specific use case of your API should be considered, ideally with team consensus.

A suitable term is "API client."

Remember, APIs describe the interaction between the client needing data/functionality and the server providing it.

Any entity requesting data from an API is an API client, regardless of the software, tool, or script used.

However, you may at times need to refer to a ‘human’ user, such as a developer making an implementation decision. In this case, “API user” may be appropriate.

If the API targets a specific client type, it's advisable to use that specific term rather than a generic one.

However, as the API expands, it may support a wider range of clients over time.

Focusing on resources

When working with APIs, interactions often revolve around resources—whether retrieving, creating, or updating them.

It's crucial to emphasize the significance of the resource object, particularly in POST requests where the newly created resource is returned in the response.

In API responses, it's essential to highlight the key fields that provide meaningful information to users.

For instance, some fields like 'created' timestamps may be less relevant for immediate user interaction.

Therefore, it's important to clarify the significant aspects of the response object and how they contribute to the overall resource representation.

Clarifying server behavior

Documenting API server behavior is essential for clarifying functionality that may not be immediately apparent to API clients.

Server behavior encompasses how an API responds to requests, often influenced by factors such as query parameters.

Identifying anti-patterns

Anti-patterns in API design refer to common pitfalls or bad practices that developers may inadvertently adopt when designing APIs.

These can include inconsistencies, inefficiencies, or ambiguities in the API structure, which can lead to confusion, frustration, and inefficiency for API consumers.

These anti-patterns can create problems when documenting an API because they may force documentation writers to clarify or explain confusing or ambiguous aspects of the API.

Additionally, documenting poor design choices may require extra effort to educate users on proper usage and workaround solutions.

Let’s look at some common anti-patterns.

Using incorrect HTTP methods or mixing actions

Issue: Using inappropriate HTTP methods for operations or combining multiple actions in one endpoint.

Example: Using POST instead of GET for retrieving data, or having an endpoint that both creates and updates resources.

Inappropriate status codes

Issue: Failing to use appropriate status codes in API responses.

Example: Returning a 200 OK status code for an error response instead of the correct 4xx or 5xx series codes.

Inconsistent naming conventions

Issue: Mixing naming conventions such as
camelCase and snake_case.

Example: userId vs user_id

Booleans as strings

Issue: Fields that should be booleans are instead strings.

Example: The acceptable field values are (0/1) instead of the appropriate boolean states (false/true).

To address these issues, writers should identify common API design flaws and provide clear, concise documentation for improved user understanding and effective API usage.

When encountering anti-patterns, writers can suggest improvements or best practices to API developers, enhancing overall quality and usability.

## Authorization Code Flow with (PKCE)

The PKCE flow enhances the security of the Authorization Code 
flow by using a Code Verifier and Code Challenge to prevent 
authorization code interception attacks. It provides a robust 
mechanism for securely handling authorization codes, especially 
in public clients, while also supporting CSRF protection.

To obtain an access token using this flow

1. Authorization Request

The API client:

- generates a Code Verifier.
- creates a Code Challenge from the Code Verifier.
- sends the user to the Authorization Server with the Code Challenge.

```shell
GET /authorize?response_type=code
&client_id=client_id
&redirect_uri=https://client.example.org/cb
&code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
&code_challenge_method=S256
&state=xyz
```

2. User Authenticates: User authenticates and authorizes the client.

3. Authorization Response: Authorization Server sends an authorization 
code to the client's redirect URI.

4. Token Request:

Client sends the authorization code, Code Verifier, and other 
required parameters to the token endpoint.

```shell
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=authorization_code
&redirect_uri=https://client.example.org/cb
&client_id=client_id
&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

5. Token Response: Authorization Server validates the Code Verifier 
against the Code Challenge and issues an access token.

## Security Best Practices

When integrating OAuth 2.0 for your applications, adhering to 
security best practices is crucial to ensure the safety and 
integrity of your system and user data. Below are key areas to focus on:

*Token Storage*

Securely storing tokens is vital to prevent unauthorized access. 
Recommendations include:

- **Mobile Apps**: Use secure storage mechanisms like iOS Keychain or 
Android Keystore.
- **Web Applications**: Store tokens in HTTP-only cookies to prevent 
access via JavaScript, mitigating XSS attacks.

*Token Rotation*

Regularly rotating tokens reduces the risk of long-term token exposure. 
Best practices include:

- **Short-Lived Tokens**: Use short expiration times for access tokens.
- **Refresh Tokens**: Implement refresh tokens with strict expiration 
policies and invalidate them if suspected of being compromised.

*Rate Limiting*

Enforcing rate limiting policies helps protect your API from abuse 
and ensures fair usage. 

Considerations include:

- **Request Limits**: Define the maximum number of requests per time unit 
(e.g., per minute, hour).
- **Penalty Mechanisms**: Implement penalties such as temporary bans for
exceeding rate limits to deter abusive behavior.

*Prevent CSRF Attacks*

Cross-Site Request Forgery (CSRF) attacks can be prevented by:

- **CSRF Tokens**: Include anti-CSRF tokens in forms and verify them 
on the server side.
- **SameSite Cookies**: Use the `SameSite` attribute for cookies to 
restrict cross-origin requests.

*Use HTTPS in Production*

Using HTTPS encrypts data in transit, preventing eavesdropping and 
man-in-the-middle attacks. 

Ensure:

- **SSL/TLS**: Always use SSL/TLS certificates for encrypting HTTP traffic.
- **Enforced HTTPS**: Redirect all HTTP requests to HTTPS and use HSTS 
(HTTP Strict Transport Security) to enforce secure connections.

Handling errors

API reference documentation focuses on crafting clear, human-readable error messages that developers can act on.

There is a particular emphasis on client errors, as these are the errors that clients can resolve.

These messages are concise and straightforward, providing just enough information for the developer to address the issue.

They do delve into the specifics of the user's context or the steps leading up to the error. v

While some brief tips on resolving the error may be included, the emphasis remains on clarity and brevity.

When more complex error handling scenarios arise, it’s a best practice to direct the reader from the API reference to the extended docs.

These types of documentation are better suited for guiding users through error resolution processes that involve more in-depth troubleshooting.

In procedural documentation, error handling is closely tied to API workflows.

As users navigate the "happy path" of the API, they may encounter errors that require them to divert to an alternative "unhappy path."

The role of procedural and conceptual documentation is to anticipate where things can go wrong.

This includes analyzing the established API flows and providing solutions or guidance when errors occur.

When users encounter the "unhappy path," the documentation should offer error handling instructions specific to the task at hand.

Or, it should refer users to a more generalized error section that discusses common error patterns and solutions.

For example, readers are following code samples and might encounter unexpected errors.

For the general error section, which focuses on common error scenarios, the goal is to provide guidance on solving frequent issues.

This involves explaining the structure and fields of error messages returned in API responses.

By defining a central error response format, you avoid duplication and ensure consistency across the documentation.

While OpenAPI’s error schema is a common standard, many APIs have their own error schemas designed to be more helpful to users of that specific API.

Conceptual and procedural documentation should address these API-specific errors, particularly those related to specific resources.

For instance, documenting a standard 404 Not Found error isn’t necessary in conceptual and procedural docs since it’s a common issue with an obvious source.

However, with a payment processing API, if a card payment is declined, it’s essential to interpret the response fields to determine the next steps.

These steps could include displaying a message to the user to contact their card issuer or attempting the payment again later.

This type of information should be centrally located so it can be referenced throughout the documentation.

Custom Error Types

After analyzing the API flows, you’ll likely notice patterns in both the structure of error messages-

And the errors encountered by specific resources or methods.

Start by identifying which resources these error types apply to.

For example, payment errors for resources involving card transactions.

Once you’ve identified the types of errors, categorize them and create centralized information on how to resolve them.

When describing a custom error type, include the following:

  • Error Code: The specific code returned when this error is encountered, along with its meaning.

When describing a custom error type, include the following:

  • Error Response Body: Provide a code sample showing the response body, and describe the error response schema specific to the custom error type.

This structured approach helps users understand, identify, and resolve API-specific errors effectively.

A typically error response has a predefined structure:

{
  "message": "Authorization AUst6ZdywPwe4fbYWrH1zTNJ was declined Cause: Authorization AUst6ZdywPwe4fbYWrH1zTNJ could not be submitted. Causes: DECLINED",
  "errors": [
    {
      "code": "DECLINED",
      "failure_message": "The card was declined for an unknown reason. The cardholder needs to contact their issuer for more information.",
      "_links": {
        "self": {
          "href": "https://api.com/authorizations"
        },
        "authorization": {
          "href": "https:/api.com/authorizations/AUst6ZdywPwe4fbYWrH1zTNJ"
        }
      }
    }
  ]
}

A typical error response in an API follows a predefined structure.

It's important to understand the fields in this response and their significance.

The top-level message field provides a general overview of the error, helping the developer identify and troubleshoot the issue.

The code field, such as DECLINED, is crucial for the API client to determine the appropriate action when this error is encountered.

The client can be programmed to display the failure_message to the end user, advising them to follow up with their financial institution.

Additionally, the response can include a link to the specific resource instance involved in the error.

The client can access the link to gain a more detailed information about that resource (e.g., an authorization).

After understanding the error response structure, outline the steps the developer should take to resolve the error or determine the next course of action.
